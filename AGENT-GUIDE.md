## 核心文档放哪

| 路径 | 用途 |
|------|------|
| **`.cursor/WORKFLOW.md`** | 阶段门禁（G0–G5）、反偷懒表、**设计还原硬约束**、自测清单、提测声明、联调、**需求变更管理**、Git 分支、**交付通告三段式**、任务清单与历史（单一真源） |
| **`<功能目录>/docs/req-spec.md`** | G0 需求规格：路由、范围、接口与验收（给参谋 / 像素侠 / 桥梁师对齐） |
| **`<功能目录>/docs/design-spec.md`** | G1 设计规格：画布、叠层、切片、`assets` 与坐标来源（给透视眼存档） |

> **功能目录**：即功能代码所在目录。页面型项目通常为 `src/pages/<模块>/<feature>/`，组件库为 `src/components/<feature>/` 等，以项目 `WORKFLOW.md`「产出文件约定」为准。同一功能的 docs / tests / e2e 聚合在该目录下。

对话里使用 **`@<功能目录>/docs/`**（如 `@src/pages/模块/功能/docs/req-spec.md`）可把上述成文说明**附加上下文**，避免只靠聊天碎片改代码。

## 门禁速查（G0 → G5）

与 `WORKFLOW.md` 一致，活动页 / 需视觉还原的需求默认不得跳步：

| 阶段 | 角色 | 产出要点 |
|------|------|----------|
| **G0** | 🧩 参谋 | `<功能目录>/docs/req-spec.md`（或书面确认「跳过」并登记） |
| **G1** | 🔍 透视眼 | `<功能目录>/docs/design-spec.md` / 蓝湖 MCP 结果**入库** |
| **G2** | 素材 | 切图进仓库约定目录；**禁止**用 1×1 占位冒充完成 |
| **G3** | ⚡ 像素侠 + 🌉 桥梁师 | 分层 + DOM 文案 + 热区 + 接口层；**自测 S1–S6 全 ✅ 方可报完成** |
| ↳ | 🎖️ 饼王 **提测声明** | 确认自测通过，正式宣告进入测试阶段 |
| **G4** | 🔎 找茬王（**强制闸**） | 测试 + UI/验收预审；不通过 → 饼王调度返工 → **复验** |
| **G5**（条件） | 🌉 桥梁师 **联调** | STUB→真实接口；完成后 **必须** 经找茬王 **联调回归** |

细节与例外见 `WORKFLOW.md` 正文。

## 饼王单入口 — 你只跟饼王说话

> **`orchestrator.mdc` 已设为 `alwaysApply: true`**，任何对话都有饼王兜底。  
> 饼王会根据你说的内容自动判断：**轻活自己做（内联）** / **重活派单到新对话** / **非工作流问题静默回答**。

                          ▼

                          │

### 你只需要会说这几句话

**新功能：**

```text
我有一个新功能要做：[需求描述]
原型图：[Figma链接 / 蓝湖 / 截图]
接口文档：[有/无]
分支：feature/[slug]

请按 WORKFLOW 开始工作流。
```

**模糊需求：**

```text
我想做一个[功能描述]，帮我先梳理需求（G0）。
```

**老项目迭代：**

```text
这是老项目迭代。
- 改动位置：[路由 / 文件 / 模块]
- 增量需求：[加什么]
- 设计稿：[有链接 / 无]
- 接口：[无变更 / 新字段 / 新接口]
- 分支：feature/[迭代slug]

请按 WORKFLOW「迭代减档」推进。
```

**小改 / Bug：**

```text
帮我把 [文件路径] 的 [xxx] 改成 [yyy]。
```

**联调：**

```text
接口文档已就绪：[链接]。请安排联调。
```

**需求变更：**

```text
需求变更：[原来是… → 现在要…]，请走变更流程。
```

**贴回 Agent 报告：**

```text
饼王，[Agent代号] 报告如下：
[粘贴完成报告]
```

**继续下一步：**

```text
继续
```

## 工作流执行流程

```
用户输入需求
     │
     ▼
🎖️ 饼王 初始化工作流（对照 .cursor/WORKFLOW.md，建 feature 分支）
     │
     ├──► 🧩 参谋 (G0) ────────────────────┐
     │         （蓝湖单链接也须先定：路由/数据/验收）│
     └──► 🔍 透视眼 (G1) ───────────────────┤
                                           │
                    ┌─── req-spec + design-spec（归档到 <功能目录>/docs/）┘
                    │
                    ▼
          G2 素材本地化（真实切图入 assets/，禁长期唯外链）
                    │
                    ▼
          ⚡ 像素侠 (G3) — 分层叠图 + DOM 文案 + 热区
                    │   （禁止整页稿单层 background；见 WORKFLOW 硬约束）
                    ├──► 可与 🌉 桥梁师 并行（视门禁）
          🌉 桥梁师 (接口/STUB)──┘
                    │
                    ▼
          📋 自测 S1–S6 全 ✅（像素侠 S1–S4 + 桥梁师 S5–S6）
                    │   ❌ → 打回修复，不得报完成
                    ▼
          🎖️ 饼王 发「提测声明」
                    │
                    ▼
          🔎 找茬王【G4 强制】测试 + UI/验收预审
                    │     不通过 ──► 饼王调度返工 → 修完 → 找茬王复验 ──┐
                    │     通过 ────────────────────────────────────────┘
                    ▼
          🛡️ 守门员 (代码审查)
                    │
                    ▼
          🎖️ 饼王 交付确认（可合并 / PR）
                    │
      ┌─────────────┴──────────────────┐
      │ 真实接口到位？                    │
      │ 是 ▼                            │ 否 → 先交付 STUB 版本
      │ 🌉 桥梁师 联调 (G5)              │     后续触发联调
      │ 🔎 找茬王 联调回归               │
      │ 🛡️ 守门员 联调 diff 审查         │
      │ 🎖️ 饼王 最终交付                 │
      └───────────────────────────────┘

      ⚠️ 需求变更？→ 饼王「变更通报」→ 更新 spec → 改码 → 找茬王变更回归
```

**Bug 修复**：修复完成后由执行方 **通知饼王**，饼王 **协调找茬王** 做回归复验，通过后再合并或交付。

### 为何会出现「没做需求分析就写页面」？

常见原因：**单轮对话里走了捷径**（只丢蓝湖链接 → 直接写页面），**未按饼王调度**走 G0/G1，也未在 `WORKFLOW.md` 登记。  
**现流程**：以 **`.cursor/WORKFLOW.md`** 为准——无 req-spec（或用户明确「跳过」并登记）不得进入 G3；蓝湖 **`detailDetach` 单图**也必须 **参谋 + 透视眼并行起手**。

### 蓝湖 / 活动页必读（通用）

1. **先 G0**：`<功能目录>/docs/req-spec.md`（须含路由 / 数据 / 验收）。
2. **再 G1**：`<功能目录>/docs/design-spec.md` 或 MCP 结果**写入仓库**，不可仅留在聊天记录。
3. **G2**：切图落在 `<功能目录>/assets/`；勿长期唯设计站外链。
4. **G3 还原**：图层 + 文案 DOM + 热区；**禁止**整稿单层 `background`。排版须对齐设计坐标 / `design-spec`，见 **`WORKFLOW.md` →「设计还原硬约束」**。

### 每轮交付你该看到什么（交付通告三段式）

阶段完成（尤其 **G3 设计落地**）时，回复里应包含：

1. **谁在执行**（Agent 代号与职能）  
2. **执行结果**（路径、构建或验收情况、遗留项）  
3. **下一步**（调度谁、你本地跑什么命令、或需要你确认什么）  

**禁止**只给代码 diff、不交代身份与下一步。模板见 **`WORKFLOW.md` →「交付通告三段式」**；饼王 **「阶段完成报告」**、像素侠 **「报告完毕」** 与之一致。

## 各 Agent 的触发时机

### 什么情况下调用参谋

- 需求模糊、只有一句话或 PRD 需结构化  
- 要定功能边界、验收标准、开放问题  

### 什么情况下可弱化参谋

- 需求与验收已在 `req-spec` 里写死，且本轮只是小迭代（仍建议在 WORKFLOW 任务清单留痕）

### 像素侠和桥梁师的并行策略

- Mock 独立时，可用两个 Cursor 对话并行；桥梁师合上后，检查是否清掉像素侠侧 `// TODO: 桥梁师` 等占位  

### 桥梁师：**没有接口文档**时做什么

- **照样接单**：不必等后端文档才开工。  
- **做法**：在 `src/api/**` 里用 **STUB**（`Promise.resolve` 写死数据 + `USE_STUB` 开关），保证 **前端主流程能演示**；注释标明 `// STUB: 待接口文档`。  
- **后续**：文档到了以后 **只改 api 层** 换成真实 `request`，减少返工。细则见 **`.cursor/rules/api-integrator.mdc`**。

### 守门员的触发时机

- 任意主要阶段完成后都可触发；合并 / 上线前建议至少完整审一轮
- 联调完成后建议再审一轮（重点：STUB 残留、鉴权、错误处理）

### 联调（G5）什么时候触发

- **不是每次都有**：仅在桥梁师用了 STUB 且真实接口后来到位时才触发。
- 饼王负责识别时机（用户说"接口好了"、或后端文档交付）并调度桥梁师。
- 联调完成后 **必须** 经找茬王联调回归，不可"改了 api 层就算完"。

### 需求变更怎么处理

- 饼王识别为变更 → 输出「变更通报」→ 参谋/透视眼更新 spec → 受影响 Agent 改码 → 找茬王变更回归。
- **禁止**直接改代码不更新 req-spec / design-spec（否则找茬王用旧 spec 验收会冲突）。
- 详见 `WORKFLOW.md` →「需求变更管理」。

### Git 分支策略

- 新功能用 `feature/<slug>` 分支，bug 修复用 `fix/<描述>` 分支。
- **合并到 main/master 前**必须经 G4（找茬王通过）+ 守门员审查 → PR。
- 联调在同一 feature 分支上继续，不另开分支。
- 具体策略以 `WORKFLOW.md`「项目信息 · Git 分支策略」为准。

## 常见问题

**Q：`<功能目录>/docs/` 是干什么用的？**  
A：放 **req-spec / design-spec** 等成文材料，**不参与前端打包**；与 `WORKFLOW` 门禁对应。对话里 `@<功能目录>/docs/` 可把这些文件当作上下文。

**Q：多个 Agent 的产出会不会互相覆盖？**  
A：约定目录不同：参谋→需求文档，透视眼→设计文档，像素侠→`src/**`，桥梁师→`src/api/**` 等；同名文件冲突时需人工或饼王协调。

**Q：如何实现 Mock / 真实接口切换？**  
A：按项目 `.env` 约定（若有 `VITE_API_MODE` 等），以本仓库实际配置为准。

**Q：守门员标了 P0 怎么办？**  
A：饼王调度对应 Agent 修复；修复后 **饼王必须再调度守门员复审** P0 条目，确认清零方可进入终态。若 P0 涉及 UI/验收类问题，需先经找茬王复验再回守门员。

**Q：日志太多怎么办？**  
A：可按里程碑归档到 `logs/archive/<日期>/`。

## MCP 配置（设计平台对接）

工作流可结合 Figma / 蓝湖 MCP；具体工具名以当前 Cursor 已配置的 Server 为准。

### 已对接的 MCP（示例）

| 平台 | 说明 |
|------|------|
| Figma | Cursor Plugin / MCP：`figma.com/design/` 等 |
| 蓝湖 | 项目内 `mcp.json` 或本地 `lanhu` / `mcp-lanhu`；需 Cookie 或团队文档 |

### 蓝湖（可选）本地参考步骤

团队若使用开源/自建 Lanhu MCP，一般步骤为：安装服务 → 配置 Cookie → 在 Cursor 中启用对应 MCP。具体命令以你们 **`.cursor/mcp.json`** 与运维文档为准（本指南不绑定单一安装方式）。

> **复用提醒**：`.cursor/mcp.json` 中的路径占位符 `<蓝湖MCP目录>` 须替换为本机实际路径（绝对或相对均可）。

### URL 与工具（示意）

| 你输入的链接 | 常见处理 |
|-------------|----------|
| `figma.com/design/...` | Figma MCP 读节点 |
| `lanhuapp.com/...stage...` | 蓝湖设计列表 / 分析 |
| `lanhuapp.com/...detailDetach...` / 含 `image_id` | 蓝湖单图画板：透视眼解析 + `design-spec` 存档 |
| `lanhuapp.com/...product...` | 多为需求文档：优先参谋 |
| 截图 / 图片 | 透视眼视觉分析（无 MCP） |

## 复用到新项目

1. 复制 **`.cursor/rules/`**  
2. 复制 **`.cursor/WORKFLOW.md`**（或团队统一改名为根目录 `WORKFLOW.md`，但须全员约定路径）  
3. 复制 **`.cursor/AGENT-GUIDE.md`**（本文件）  
4. 按需复制 **`.cursor/mcp.json`**、建立 **`logs/`** 目录  
5. 在新仓库 **「项目信息」** 表（`WORKFLOW.md`）里填写技术栈与静态资源约定  
6. 开启对话时使用 **@饼王 / @orchestrator** 并按 **交付三段式** 验收每阶段收尾
