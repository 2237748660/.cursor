## 饼王单入口模式

> 用户只跟饼王对话；饼王负责调度或内联执行。

| 情况 | 饼王行为 |
|------|----------|
| 轻活（<=3 文件、<=50 行、无视觉还原） | 直接内联执行 |
| 重活（整页、API 层、测试套件、CR） | 输出派单指令 |
| 并行重活 | 输出多份派单指令并行推进 |
| 非工作流问题（解释代码/问语法） | 静默直答，不走调度 |

## 强制阶段顺序与门禁（G0-G5）

| 序号 | 阶段 | 准入条件 | 产出 |
|------|------|----------|------|
| G0 | 🧩 需求分析 | 饼王接收需求 | `<功能目录>/docs/req-spec.md` |
| G1 | 🔍 设计分析 | G0 完成或书面跳过 | `<功能目录>/docs/design-spec.md` |
| G2 | 素材落地 | G1 可用 | 切图本地化到仓库约定路径 |
| G3 | ⚡ UI 开发 | G0+G1+G2 满足 | 分层页面（底图/文案/热区） |
| G4 | 🔎 测试与验收预审（强制） | G3 自测通过 + 提测声明 | 测试结果 + UI/验收结论 |
| G5 | 🌉 联调（条件触发） | 真实接口文档就绪 | STUB->真实接口 + 回归通过 |

**例外规则**：用户明确要求“跳过需求直接实现”时，须登记到「待确认项」，并至少补一页最小 req-spec（路由、接口、验收<=5条）。

### 迭代减档（老项目迭代 / 小改 / Hotfix）

对于改动范围有限的迭代场景，饼王可根据实际情况**合并或简化**部分阶段，但 **G4（找茬王）不得跳过**：

| 场景 | 可简化阶段 | 仍强制 | 说明 |
|------|-----------|--------|------|
| **小迭代**（增量需求，改动 ≤ 3 页面） | G0 可内联（饼王在任务清单补最小 req-spec）；G1 无新设计稿时可跳过 | G3 自测 + G4 找茬王 | 需在任务清单登记「迭代减档」及跳过理由 |
| **Bug 修复 / Hotfix** | G0–G2 可全部跳过 | 修复后必须通知饼王 → 饼王调度找茬王做回归复验 | 修复分支用 `fix/<issue>` |
| **纯文案/配置调整**（≤ 3 文件 ≤ 50 行） | G0–G2 跳过；饼王内联执行 | 构建通过即可；若涉及用户可见文案变更，仍建议找茬王抽检 | 饼王在日志登记 |
| **接口无变化的 UI 微调** | G0–G2 跳过；桥梁师不参与 | G3 自测 S1–S4 + G4 找茬王 | 设计稿有变则 G1 仍需更新 design-spec |

**核心原则**：减档 ≠ 跳过质量关卡。任何涉及主流程或用户可见行为的改动，**找茬王回归不可省略**。饼王须在任务清单「状态」列标注 `减档` 并注明跳过了哪些阶段。

## 反偷懒规范（强约束）

| 违规形态 | 流程要求 |
|----------|----------|
| 未做需求分析就写页面 | 必须先完成 G0 或登记跳过原因 |
| 整页图当唯一背景 | 禁止；必须分层可维护 |
| 不切图长期外链 | 必须 G2 本地化 |
| 单会话一步到位 | 每阶段需有阶段报告 |
| 占位图冒充切图 | 禁止；需真实可辨资源 |
| 无坐标凭感觉排版 | 需按标注或在 spec 说明换算 |
| 结构臆造 | 以图层树/MCP 结构为准 |
| 文案与稿/req 不一致 | 统一以稿和 req 为准 |
| 热区被装饰层盖住 | 热区必须在可点最上层 |
| 无文档不接活 | 可先 STUB 打通，后续 G5 联调替换 |
| 跳过找茬王直接完成 | 禁止；G4 必走 |
| 联调后不回归 | 禁止；必须回归 |
| 变更不更新文档 | 禁止；req/design-spec 必同步更新 |

## 设计还原硬约束（G3 交付标准）

| 根因 | 禁止 | 正确做法 |
|------|------|----------|
| 占位素材 | 1x1 或无意义占位图 | 真实导出/MCP资源，具真实宽高 |
| 结构错误 | 臆造流式结构 | 按设计层级还原（流式 vs 绝对） |
| 坐标漂移 | 手写大概百分比 | 统一标尺换算并落到 spec |
| 底图缺失 | 仅零散切片 | 明确主底图与 z-index 顺序 |
| 文案漂移 | 与稿件不一致 | 对齐 req + 设计文案 |
| 交互层错误 | 热区不可点 | 提升 z-index 并可验证 |

**G3 完成前置清单（S1-S6）**

1. S1 构建通过（`pnpm build` 或团队等价命令）。
2. S2 路由可达，无白屏无报红。
3. S3 主流程可点通一遍。
4. S4 设计并排抽查 >= 1 屏。
5. S5 接口层可运行（STUB/Mock/真实均可）。
6. S6 无散落假数据（统一从 `src/api` 获取）。

## 交付通告三段式（所有 Agent 报告通用）

每个 Agent 阶段完成时（尤其 G3 设计落地、找茬王验收、守门员终裁），回复/日志中 **必须** 包含以下三段，禁止只给代码 diff 不交代身份与下一步：

1. **谁在执行**（Agent 代号与职能）
2. **执行结果**（产出路径、构建/验收情况、遗留项/P0）
3. **下一步**（调度谁、用户本地跑什么命令、或需要用户确认什么）

## 找茬王阶段（G4 · 强制闸）

在 G3（像素侠）与本功能的接口层（桥梁师）均可构建、可演示之后，**必须** 进入找茬王：

1. **测试**：按项目已有约定补齐或执行单测 / E2E / 接口测等（无框架时至少在日志中列出手工用例与结果）。
2. **UI/验收预审**：对照 **设计还原硬约束**、`req-spec`、`design-spec`（热区、文案、占位图、叠层）；结论写入 `logs/找茬王 — 测试工程师.log`。
3. **不通过**：找茬王出具 **驳回报告** -> 饼王调度返工 -> 修复完成后 **饼王必须再次调度找茬王复验**，闭环后再进入守门员或交付通告。

> **与守门员**：找茬王侧重 **可验证的验收与稿面一致性**；守门员侧重 **代码质量与安全**。默认 **找茬王通过后再** 进行守门员终裁。

## 联调阶段（G5 · 条件触发）

> 仅在桥梁师此前使用了 STUB / Mock 且真实接口文档到位时触发。若一开始就对真实接口，则跳过 G5。

### 触发时机

以下任一条件满足时，饼王 **必须** 调度联调：

1. 后端接口文档交付（Swagger / OpenAPI / 手写）。
2. 后端已部署可联调的测试环境。
3. 用户在对话中说「接口好了 / 可以联调了」。

### 联调流程

| 步骤 | 角色 | 动作 |
|------|------|------|
| 1 | 🌉 桥梁师 | 关闭 `USE_STUB`；替换为真实 `request`；逐接口核对字段名、状态码、分页、错误码 |
| 2 | 🌉 桥梁师 | **联调自测**：构建通过 + 主流程真实接口跑通；报告列出「已对齐 / 待后端修复 / 前端需调整」 |
| 3 | ⚡ 像素侠（若需） | 因接口返回结构变化导致的 UI 适配 |
| 4 | 🔎 找茬王 — 联调回归 | **必须** 重跑 G4 P0 用例 + 接口用例；确认真实数据下无回退 |
| 5 | 🛡️ 守门员（建议） | 审查 STUB → 真实接口 diff，确认无残留 STUB、鉴权正确 |

### 联调完成标准

- 桥梁师报告中 **无「待后端修复」阻塞项**。
- 找茬王 **联调回归通过**。
- `src/api` 中 **无残留 `USE_STUB = true`**（或仅保留显式注释开关）。

## 产出文件约定（按功能目录聚合）

> **核心原则**：同一功能的 docs、tests、e2e 放在功能目录下，而非散落到不同顶层文件夹。  
> **功能目录**：指功能代码所在的目录。页面型项目通常是 `src/pages/<模块>/<feature>/`；组件库可能是 `src/components/<feature>/`；工具库可能是 `src/<feature>/`。以项目实际结构为准，在「项目信息」表中填写约定。

```
<功能目录>/                   # 按项目约定（示例：src/pages/<模块>/<feature>/）
├── index.vue / index.tsx    # 页面入口（文件类型随技术栈）
├── *.vue / *.tsx            # 子页面/组件
├── assets/                  # 切图与静态资源
├── docs/                    # 需求规格 + 设计规格
│   ├── req-spec.md
│   └── design-spec.md
├── tests/                   # 单元测试 + API 测试
│   └── *.spec.{ts,js}
└── e2e/                     # E2E / 流程测试
    └── *.spec.{ts,js}
```
