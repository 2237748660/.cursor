# Cursor 多 Agent 协同工作流（通用模板）

> 一套即拿即用的 **Cursor Rules + 工作流引擎**，让 AI 按「需求分析 → 设计解析 → UI 开发 → 接口集成 → 测试验收 → 代码审查」的标准化流水线协作，适配任意前端技术栈。

---

## 快速开始

### 1. 复制到你的项目

```bash
# 在目标项目根目录下
cp -r <本模板路径>/.cursor .cursor
```

需要复制的文件：

```
.cursor/
├── rules/                  # 7 个 Agent 规则
│   ├── orchestrator.mdc    # 🎖️ 饼王 — 中控调度（alwaysApply）
│   ├── req-analyst.mdc     # 🧩 参谋 — 需求分析
│   ├── design-analyst.mdc  # 🔍 透视眼 — 设计解析
│   ├── ui-builder.mdc      # ⚡ 像素侠 — UI 开发
│   ├── api-integrator.mdc  # 🌉 桥梁师 — 接口集成
│   ├── test-writer.mdc     # 🔎 找茬王 — 测试 + 验收
│   └── reviewer.mdc        # 🛡️ 守门员 — 代码审查
├── WORKFLOW.md             # 工作流母板（门禁 / 任务清单 / 规范）
├── AGENT-GUIDE.md          # 面向人的使用指南
├── mcp.json                # MCP 配置（蓝湖等，按需修改）
└── README.md               # 本文件
```

### 2. 填写项目信息

打开 `.cursor/WORKFLOW.md`，填写「项目信息（启动必填）」表：

| 字段 | 说明 |
|------|------|
| 项目名称 | 你的项目名 |
| 技术栈 | Vue3 / React / Next / uni-app 等 |
| 接口模式 | 真实接口 / STUB / Mock |
| 原型来源 | Figma / 蓝湖 / 截图 / 无 |
| 静态资源约定 | 功能目录下 assets 的组织方式 |
| Git 分支策略 | 如 `feature/<slug>` 从主分支拉取 |

### 3. 配置 MCP（可选）

如果使用蓝湖，编辑 `.cursor/mcp.json`，将 `<蓝湖MCP目录>` 替换为本机实际路径。Figma MCP 通过 Cursor 插件自动可用。

### 4. 开始对话

在 Cursor 中开启新对话，直接跟饼王说话即可：

```text
我有一个新功能要做：[需求描述]
原型图：[Figma链接 / 蓝湖 / 截图]
接口文档：[有/无]
分支：feature/[slug]

请按 WORKFLOW 开始工作流。
```

## 核心概念

### 饼王单入口

**你只跟饼王说话**。`orchestrator.mdc` 设为 `alwaysApply: true`，任何对话都有饼王兜底。饼王根据任务复杂度自动决定：

- **轻活**（≤ 3 文件 / ≤ 50 行）→ 内联执行
- **重活**（整页 / API 层 / 测试套件）→ 输出派单指令，你新开对话执行
- **非工作流问题** → 静默直答

### 阶段门禁（G0–G5）

```
G0 需求分析 → G1 设计分析 → G2 素材落地 → G3 UI+接口开发 → G4 测试验收 → G5 联调（条件）
```

- 每个阶段有准入条件和产出物要求
- G4（找茬王）为**强制闸**，不可跳过
- G5 仅在有 STUB 且真实接口到位时触发

### 迭代减档

对改动范围有限的场景（小迭代 / Bug修复 / 纯文案），前置阶段可简化，但 G4 找茬王仍强制。详见 `WORKFLOW.md`「迭代减档」。

## 7 个 Agent

| 代号 | 角色 | 规则文件 | 核心职责 |
|------|------|---------|---------|
| 🎖️ **饼王** | 中控大脑 | `orchestrator.mdc` | 全局调度、状态维护、门禁执行 |
| 🧩 **参谋** | 需求分析 | `req-analyst.mdc` | 需求拆解 → `req-spec.md` |
| 🔍 **透视眼** | 设计解析 | `design-analyst.mdc` | Figma/蓝湖/截图 → `design-spec.md` |
| ⚡ **像素侠** | UI 开发 | `ui-builder.mdc` | 分层还原 + 自测 S1–S4 |
| 🌉 **桥梁师** | 接口集成 | `api-integrator.mdc` | API/STUB + 自测 S5–S6 |
| 🔎 **找茬王** | 测试验收 | `test-writer.mdc` | 单测/E2E + UI验收预审（**强制闸**） |
| 🛡️ **守门员** | 代码审查 | `reviewer.mdc` | 代码质量 + 跨 Agent 衔接检查 |

                    ▼

## 产出文件约定

同一功能的所有产出聚合在功能目录下：

```
<功能目录>/
├── index.vue / index.tsx      # 页面入口
├── assets/                    # 切图与静态资源
├── docs/
│   ├── req-spec.md            # 需求规格（参谋）
│   └── design-spec.md         # 设计规格（透视眼）
├── tests/
│   └── *.spec.{ts,js}         # 单元/API 测试（找茬王）
└── e2e/
    └── *.spec.{ts,js}         # E2E 测试（找茬王）
```

> **功能目录**：页面型项目通常为 `src/pages/<模块>/<feature>/`；组件库为 `src/components/<feature>/` 等，以 `WORKFLOW.md` 约定为准。

## 关键机制

### 交付三段式

每个 Agent 报告必须包含：**谁在执行** → **执行结果** → **下一步**。

### 驳回与复验闭环

- **找茬王不通过** → 饼王调度返工 → 修复后**必须找茬王复验**
- **守门员 P0** → 饼王调度修复 → 修复后**必须守门员复审**
- 禁止跳过复验/复审直接标记完成

### 需求变更管理

饼王识别变更 → 变更通报 → 更新 spec → 改码 → 找茬王变更回归

### STUB 策略（无接口文档时）

桥梁师用 `Promise.resolve` 写死数据 + `USE_STUB` 开关，保证前端可演示。接口到位后仅改 `src/api` 层。
